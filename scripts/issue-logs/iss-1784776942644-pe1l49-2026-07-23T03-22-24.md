## Summary

### Root Cause
The assistant was text-only — it rendered Markdown in chat bubbles but had no mechanism to speak responses aloud. No TTS, audio streaming, or speech synthesis infrastructure existed anywhere in the codebase. When users asked the assistant to speak, nothing happened.

### Diagnosis
After examining the full assistant stack (`AssistantBar.tsx`, `assistant.ts` server route, `api.ts`, `types.ts`, `icons.tsx`, `styles.css`), I found:
- The `AssistantBar` component renders assistant messages as React-Markdown with only a copy-to-clipboard button
- No speech/voice/TTS code existed in the frontend, server, or CSS
- Modern browsers ship a free, local Web Speech API (`SpeechSynthesis`) that provides exactly what's needed — voice selection, streaming-like playback, rate/pitch control — with no server-side changes or API keys required

### Changes Made

**`web/src/icons.tsx`** — Added three FontAwesome icons:
- `faVolumeHigh` → `speak` (speaker icon for the "read aloud" button)
- `faStop` → `speak-stop` (stop icon, shown while speaking)
- `faVolumeXmark` → `mute` (available for future use)

**`web/src/components/AssistantBar.tsx`** — Added full TTS capability:
- **State**: `speakingMessageIndex`, `availableVoices`, `selectedVoiceUri`, `voiceMenuOpen`, plus refs for the utterance and voice menu DOM node
- **Voice loading**: Hydrates available synthesis voices on mount via `voiceschanged` event, defaults to the user's locale language, persists the last-chosen voice to `localStorage`
- **`speakMessage(index, text)`**: Strips markdown formatting from the message text (code blocks, links, bold/italic, headings, lists, blockquotes), creates a `SpeechSynthesisUtterance`, attaches the selected voice, and plays it. Clicking the same message again stops playback.
- **`stopSpeaking()`**: Cancels any in-progress speech and resets state
- **Click-outside handler**: Closes the voice dropdown when clicking elsewhere
- **UI**: Voice selector dropdown (volume icon) in the session bar with a scrollable menu of available voices; speak/stop button on each assistant message bubble next to the copy button
- **Session reset integration**: `stopSpeaking()` is called on `resetToNewSession()`

**`web/src/styles.css`** — Added 60 lines of CSS:
- `.assistant-voice-select` / `.assistant-voice-menu` — positioned dropdown for voice selection with the project's existing design tokens (border, shadow, accent colors)
- `.assistant-voice-menu__head` — uppercase section header
- `.assistant-voice-menu__item` — voice option rows with hover and active states
- `.assistant-log__speak` — speak button positioned left of the copy button, hidden by default, revealed on hover (matching the copy button's behavior)
- `.assistant-log__speak--active` — persistent visible state with accent color when actively speaking

### Design decisions
- **Zero-server-change**: Used the browser-native Web Speech API, avoiding TTS API keys, server-side audio generation, streaming infrastructure, or additional dependencies
- **Graceful degradation**: If `speechSynthesis` isn't available (headless browsers, very old environments), the feature simply doesn't render — all existing functionality is untouched
- **Matches existing UI patterns**: The speak button follows the exact same hover-reveal pattern as the copy button; the voice menu follows the sessions panel styling; all colors/spacing use the existing CSS custom properties
