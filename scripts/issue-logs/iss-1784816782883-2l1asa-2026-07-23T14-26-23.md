### Details

  what: Buckets have no protection mechanism analogous to the container `protected` flag. The user wants to guard critical buckets (like dockyard-knowledge) from accidental deletion.
  expected: A `protected` boolean on bucket creation plus an update path to toggle it. When a bucket is protected: delete_bucket and delete_bucket_object are rejected with a clear error. Writes (write_bucket_object, write_bucket_objects, replace_in_bucket_object) continue to work normally — protection is deletion-only.
  context: The container system already has this exact feature (protected flag, guard logic, UI toggle, assistant tool support), so much of the pattern can be borrowed. Effort estimate is small-to-medium, on the order of a few days.

## Summary: Bucket-level protection flag

The issue was that buckets had no protection mechanism to prevent accidental deletion — a gap compared to containers which already had this feature. The fix mirrors the container `protected` flag pattern across the full stack.

### What was changed

**Database (`server/src/db.ts`):**
- Added a `protected` column to the `bucket_owners` table (migration — safe for existing deployments)
- Added `isBucketProtected(name)` and `setBucketProtected(name, protect)` functions
- Updated `setBucketOwner()` to accept an optional `protect` parameter

**Server routes (`server/src/routes/buckets.ts`):**
- `POST /` — now accepts `{ name, protected?: boolean }`; passes protection flag to `setBucketOwner`
- `DELETE /:name` — rejects with 403 if the bucket is protected ("unprotect it before deleting")
- `DELETE /:name/objects/:key` — rejects with 403 if the bucket is protected ("unprotect it before deleting objects")
- `GET /:name` — new endpoint returning bucket metadata including `protected` status
- `POST /:name/protected` — new toggle endpoint accepting `{ protected: boolean }`
- List endpoint now includes `protected` in each bucket's response
- Write operations (`PUT`, `POST /objects/bulk`, `POST /objects/replace`) are **unchanged** — protection is deletion-only

**Assistant tools (`server/src/routes/assistant.ts`):**
- `create_bucket` — now accepts an optional `protected` parameter
- `update_bucket` — new tool toggling the protected flag on a bucket
- `delete_bucket` / `delete_bucket_object` — descriptions updated to mention the protection guard
- `list_buckets` read-only handler now includes `protected` in its response
- System prompt updated to recommend protecting critical buckets like `dockyard-knowledge`

**Frontend types (`web/src/types.ts`):**
- Added `protected?: boolean` to the `Bucket` interface

**Frontend API (`web/src/api.ts`):**
- `bucketCreate(name, protect)` — updated to accept optional protect boolean
- `bucketGet(name)` — new method for single-bucket metadata
- `bucketUpdateProtected(name, protect)` — new method for toggling protection

**Frontend UI (`web/src/components/BucketPanel.tsx`):**
- **BucketList**: protected buckets show a 🔒 indicator
- **BucketDetail**: a protect/unprotect toggle button in the header; delete attempts show a toast error when the bucket is protected (both client-side guard and server-side enforcement)

**Frontend AssistantBar (`web/src/components/AssistantBar.tsx`):**
- `update_bucket` action label and `runAction` handler added
- `create_bucket` handler updated to pass the `protected` parameter
