## Response

## Summary

### Root Cause

In `scripts/issue-consumer.mjs:375`, the `pushToGitHub` function stages changes with:

```
git add -A ':!data/github-repos'
```

The `-A` flag stages **all** files — modified, deleted, and untracked — across the entire working tree. The only exclusion was `data/github-repos`. When SQLite database files like `data/iaas.db-wal` and `data/iaas.db-shm` were present (because the app was running), the consumer's commit included them alongside legitimate code changes. GitHub's push protection then blocked the push because secret scanning flagged the WAL file.

There was no `.gitignore` at the repo root, and no exclusions for database artifacts in the `git add` command, so nothing prevented this.

### Changes Made

**1. `scripts/issue-consumer.mjs:375`** — Updated the `git add` command to explicitly exclude SQLite database files:

```diff
- git add -A ':!data/github-repos'
+ git add -A ':!data/github-repos' ':!data/*.db' ':!data/*.db-wal' ':!data/*.db-shm' ':!data/*.db-journal'
```

This prevents the consumer from ever staging `.db`, `.db-wal`, `.db-shm`, or `.db-journal` files, even if they're already tracked by git.

**2. `.git/info/exclude`** — Added the same exclusion patterns at the git level (local clone only, not committed). This mirrors what a root `.gitignore` would do — the root `.gitignore` couldn't be created because the `/app` root filesystem is read-only for new file creation.

### Why Two Layers?

- The **`git add` exclusion** in the consumer is the direct fix — it stops the consumer from committing database files regardless of git tracking state.
- The **`.git/info/exclude`** patterns provide a git-level safety net for any `git add` operations that happen outside the consumer.
