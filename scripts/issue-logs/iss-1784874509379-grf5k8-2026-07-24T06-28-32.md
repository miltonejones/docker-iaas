## Diagnosis

### Root cause

The `GatewayList` component in `GatewayPanel.tsx` had **no pagination** on the grouped route names list. It rendered all groups in a single, unlimited list — whether in table or grid view. As the number of gateway route names grows, the list becomes unwieldy with no way to navigate through pages of results.

The server side (`db.ts:319-324`, `gateway.ts:107-114`) returns all routes without a `LIMIT`, which is fine for correctness — the full data set is available. But the UI had no mechanism to paginate the client-side grouped view, so any expansion beyond a single screen's worth of entries had no user-facing navigation.

### What was changed

**`/app/web/src/components/GatewayPanel.tsx`** — the `GatewayList` component:

1. **Added pagination state** (lines 113-122): Introduced `page`, `totalPages`, `effectivePage`, and `paginatedGroups`. The page size is 10 items. An `effectivePage` variable clamps the page number when the underlying list shrinks (e.g., after a deletion on the last page). A `useEffect` resets to page 0 whenever the number of routes changes.

2. **Changed rendering to use paginated groups**: Both the table view (`groups.map` → `paginatedGroups.map`) and the grid view use the paginated slice.

3. **Added pagination controls** (lines 309-328): A "← Prev / Page X of Y / Next →" bar appears below the list when there is more than one page. The Prev/Next buttons disable at the boundaries.

The CSS classes `.pagination` and `.pagination__info` were already present in `styles.css` (lines 3696-3706), so no CSS changes were needed.

### How this fixes the issue

With 15 route entries across 10 unique route names, the Gateway List now shows up to 10 grouped routes per page with Prev/Next navigation. If a user has 5 route names, all fit on one page (no pagination bar). At 11+ route names, the pagination bar appears, and users can click through to see all their routes.
