## Root Cause

The delete button in the sessions offcanvas panel (`App.tsx:500-506`) is rendered with classes `btn btn--ghost btn--sm offcanvas-session-row__delete`. While the original CSS at `styles.css:2899-2902` did have `border: none` via a `.offcanvas-session-row__delete.btn` selector, it was fragile due to competing specificity:

1. **`.btn`** (line 1009, specificity `0,1,0`) sets `border: 1px solid var(--border-strong)` — a full border shorthand.
2. **`.btn--ghost`** (line 1055, specificity `0,1,0`) sets `border-color: var(--border)`, overriding only the color. The `1px solid` from `.btn` remains.
3. **`.offcanvas-session-row__delete.btn`** (line 2899, specificity `0,2,0`) sets `border: none`, which should override both. However, **`.btn:hover:not(:disabled)`** (line 1026, specificity `0,3,0`) sets `border-color: var(--muted-2)` on hover — this has **higher specificity** than the delete rule and can reintroduce border-color on hover, making the border visible again depending on how the browser resolves the shorthand/longhand cascade.

The combination creates a situation where a border can appear, especially on hover.

## Changes (`web/src/styles.css`)

1. **Simplified the selector** from `.offcanvas-session-row__delete.btn` to `.offcanvas-session-row__delete` — the delete button's styling shouldn't depend on `.btn` being present.
2. **Added `!important`** to `border: none` to guarantee it wins against ALL competing border declarations from `.btn`, `.btn--ghost`, and their hover/focus states.
3. **Added `outline: none`** to prevent the browser's default focus outline from appearing as a border-like visual ring.
4. **Added an explicit hover rule** (`.offcanvas-session-row__delete:hover:not(:disabled)`) with `border: none !important` to ensure the hover state never re-introduces a border.
