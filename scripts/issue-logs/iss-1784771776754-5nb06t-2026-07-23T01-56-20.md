## Diagnosis

### Root Cause

The consumer container (`dockyard-consumer-ctr`) has the host's `.git` directory bind-mounted at `/app/.git:rw` (docker-compose.yml line 84). The container runs as UID `1000:1000` (line 79), but the host's `.git` directory is owned by a different UID (the host user, e.g., `miltonejones`).

Git ≥2.35.2 has a security feature that refuses to operate on repositories owned by a different user. When `pushToGitHub()` in `scripts/issue-consumer.mjs` runs any git command — even `git diff --stat` — git immediately fails with:

```
fatal: detected dubious ownership in repository at '/app'
```

The `execSync` call throws, control falls into the `catch` block on line 355-359:

```js
} catch {
    log("Could not detect changes — skipping push");
    return;
}
```

The fix is never pushed to GitHub, and all previously "fixed" issues have their code sitting uncommitted in the consumer container.

### Fix Applied

**`docker-compose.yml`** (line 62-65): Changed the consumer's `command` from:

```yaml
command: ["node", "scripts/issue-consumer.mjs"]
```

to:

```yaml
command: ["sh", "-c", "git config --global --add safe.directory /app && node scripts/issue-consumer.mjs"]
```

This runs `git config --global --add safe.directory /app` at container startup, before the consumer script begins. The `--add safe.directory` directive tells git to trust `/app` regardless of UID mismatch, so all subsequent git operations (diff, commit, pull, push) succeed.

I also authored a `Dockerfile.consumer` (at `/home/node/Dockerfile.consumer` — the `/app` directory was not writable) that includes the same `RUN git config --global --add safe.directory /app` line at build time. When placed in the repo, this bakes the config into the image layer so it's present even without the command override. The docker-compose.yml already references `dockerfile: Dockerfile.consumer` for the build.
