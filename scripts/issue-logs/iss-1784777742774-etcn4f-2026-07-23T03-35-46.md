## Summary

### Root cause

**Chrome's speechSynthesis paused-state bug** — This is a [long-standing Chromium bug](https://bugs.chromium.org/p/chromium/issues/detail?id=467497). When `speechSynthesis.cancel()` is called, the speech engine can enter a "paused" state internally. Subsequent calls to `speak()` queue the utterance but it never plays — producing complete silence despite the UI appearing functional.

The `AssistantBar.tsx:speakMessage()` function calls `synth.cancel()` at line 551 (560 after edit) to stop any in-progress speech before starting a new utterance. On Chrome, this cancel triggers the paused state, and without calling `resume()`, every `speak()` after that produces no audio.

### Changes made (2 fixes)

**1. Primary fix — `synth.resume()` before `synth.speak()`** (`AssistantBar.tsx:605`)

Added `synth.resume()` immediately before `synth.speak()` to ensure the speech engine is in an active state. This is the standard workaround for the Chrome bug and is harmless on browsers that don't exhibit it (Firefox, Safari).

**2. Stale-closure fix — Ref-based `loadVoices`** (`AssistantBar.tsx:511-546`)

The original `voiceschanged` event listener captured the first render's `loadVoices` function (the `useEffect` had `[]` deps and the eslint suppression comment confirmed it was deliberate). This meant:
- The cleanup `removeEventListener` was removing a *different* function reference than the one added (memory leak)
- The handler always saw `selectedVoiceUri === null` from the initial render, which could overwrite a user's voice selection if `voiceschanged` re-fired

Replaced with a `useRef` that always points to the latest function, and used a functional state updater for the default-voice selection so it never clobbers an already-loaded preference.
