## Summary

### Root Cause

The session bleed bug was in **`web/src/App.tsx`**. Two event handlers — `submitAssistantQuery` (toolbar search) and the `onOpenAssistant` listener (detail-page "explain this" button) — were not clearing the `assistantSessionId` state before opening a new assistant prompt.

**Exact failure sequence:**

1. The pinned assistant panel is open with session A loaded. `assistantSessionId` = session A's ID.
2. User types a **new** query in the toolbar and presses Enter → `submitAssistantQuery()` fires.
3. `modalKey` increments, forcing an `AssistantBar` remount, but **`assistantSessionId` is not cleared**.
4. The new `AssistantBar` receives `initialSessionId={assistantModal.sessionId ?? assistantSessionId}` = session A's ID.
5. `AssistantBar`'s mount effect loads session A's **entire chat history** (messages, log, pending) into state.
6. The `initialPrompt` effect sees `initialSessionId` is set and **skips** the new prompt: `if (initialSessionId) return;`
7. Result: the user sees old chat from a different session, and their new query is silently dropped.

The same sequence occurs when a detail page emits `emitOpenAssistant(prompt)` — e.g., clicking an info icon on a function or container.

### What I Changed

**`web/src/App.tsx`** — two edits:

1. **Line 284** (`submitAssistantQuery`): Added `setAssistantSessionId(undefined)` before `setAssistantModal({ prompt: q })`, so the toolbar search no longer leaks the previous session ID into the new prompt.

2. **Line 276** (`onOpenAssistant` handler): Added `setAssistantSessionId(undefined)` before `setAssistantModal({ prompt })`, so detail-page "explain" buttons no longer leak the previous session ID.
