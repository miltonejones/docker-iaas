## Root Cause Analysis

The TTS `speakMessage` function in `AssistantBar.tsx` had no safety net for silent failures. The core issue was:

**Primary cause: Chrome loads voices asynchronously.** After a hard refresh (or first visit), `window.speechSynthesis.getVoices()` returns `[]` until the `voiceschanged` event fires. Calling `speak()` with no voices available silently queues the utterance — it never fires `onstart`, `onend`, or `onerror`. The user hears nothing and sees nothing.

**Secondary causes:**
- No timeout mechanism if `onstart` never fires
- No post-`speak()` validation to detect when the browser silently rejects the utterance
- `onerror` handler just reset state without showing any error to the user
- No `try/catch` around `speak()`

## Changes Made

### `/app/web/src/components/AssistantBar.tsx`

1. **Added `ttsError` state and `ttsTimeoutRef` ref** (lines 292-293) — new state for visible TTS error feedback and a ref for the watchdog timeout.

2. **Rewrote `speakMessage()` with 5 guards** (lines 551-699):
   - **Guard 1 — Browser support**: Checks `window.speechSynthesis` exists before proceeding.
   - **Guard 2 — Voice availability**: Checks `synth.getVoices().length === 0` at speak time. If no voices are loaded (Chrome's async loading), shows: *"No voices available. Voices may still be loading — please try again in a moment."*
   - **Guard 3 — Try/catch**: Wraps `synth.speak()` in try/catch for defense-in-depth.
   - **Guard 4 — Post-speak validation**: Uses `queueMicrotask` to check `synth.speaking` and `synth.pending` immediately after `speak()`. If both are false, the browser silently blocked it — shows: *"Speech synthesis was blocked by the browser..."*
   - **Guard 5 — Onstart timeout**: Sets a 500ms watchdog. If `onstart` hasn't fired by then, cancels the utterance and shows: *"Speech synthesis timed out waiting to start..."*
   - **Improved `onerror` handler**: Now reads `event.error` and surfaces it (except for user-initiated `canceled`/`interrupted`).

3. **Updated `stopSpeaking()`** (lines 702-711) — clears the timeout on manual stop to prevent stale error messages.

4. **Updated `resetToNewSession()`** — clears `ttsError` on session reset.

5. **Added error banner UI** (lines 1494-1505) — visible in the assistant session bar, with a dismiss button. Uses `role="alert"` for screen reader accessibility.

### `/app/web/src/styles.css`

6. **Added `.assistant-tts-error` and `.assistant-tts-error__dismiss` styles** (lines 1384-1413) — red banner styling consistent with error conventions, with a right-aligned dismiss button.
