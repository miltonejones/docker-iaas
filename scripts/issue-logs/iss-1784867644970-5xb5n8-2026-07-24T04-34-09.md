## Response

## Summary

### Root Cause

The Issues list page (`IssuesList.tsx`) displayed all issues in a single unbroken table with no pagination. When many issues accumulate (the server already limits to 50), the page becomes unwieldy — long scroll, hard to navigate. There was no pagination infrastructure anywhere in the frontend.

### What I Changed

**`/app/web/src/pages/IssuesList.tsx`** — Added client-side pagination:

- Added `PAGE_SIZE = 10` (10 items per page)
- Added `page` state (0-indexed), reset to 0 when the status filter changes
- Computes `paginatedIssues` via `useMemo` to slice the full issue list to the current page
- Clamps the page index if the filtered list shrinks (e.g. switching from "All" to a sparse status like "Deploying")
- Pagination controls (← Previous / Page X of Y / Next →) appear below the table only when there are multiple pages
- Previous/Next buttons disable at page boundaries

**`/app/web/src/styles.css`** — Added minimal pagination styles:

- `.pagination` — centered flex row with 12px gap
- `.pagination__info` — muted 13px text for the page counter

### Design Decisions

- **Client-side pagination** was chosen over server-side because the server already returns a bounded set (50 issues max), and client-side pagination is simpler, requires no API changes, and gives instant page transitions
- The page resets to 0 when the status filter changes, so the user always starts at the first page of a new filter
- The pagination bar only renders when `totalPages > 1` — no UI clutter when there are ≤10 issues
