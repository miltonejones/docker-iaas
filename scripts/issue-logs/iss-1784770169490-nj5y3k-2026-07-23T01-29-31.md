## Diagnosis

**Root cause:** The assistant's system prompt (`SYSTEM` constant in `server/src/routes/assistant.ts:98`) had no mention of the Dockyard issue consumer — what it is, what it does, or when to use the three consumer-interaction tools. The tools themselves (`get_consumer_status`, `get_consumer_activity`, `retry_issue`) were already defined in the tool list (lines 1013-1038) and marked as read-only in the auto-resolve set (lines 1067-1068), but without prompt guidance the assistant had no reason to reach for them.

This meant:
- After calling `report_issue`, the assistant would just say "done" and move on
- The user got no visibility into whether the consumer picked up the issue
- The auto-fix feedback loop (issue logged → consumer fixes → commit pushed → issue resolved) was invisible to both assistant and user

## Change

Added a new rule to the `SYSTEM` prompt at `server/src/routes/assistant.ts:125` that:

1. **Explains what the consumer is** — an auto-fix bot that polls the issue store, applies Claude Code to diagnose/fix issues against the codebase, and pushes commits to GitHub
2. **Tells the assistant when to use each tool:**
   - `get_consumer_status` — check if the consumer is idle or actively working on a specific issue
   - `get_consumer_activity` — review recent fix attempts, outcomes, and GitHub commit links
   - `retry_issue` — re-open an issue the consumer failed to process
3. **Emphasizes the feedback loop** — the assistant should proactively surface when the consumer resolves an issue, since the user wouldn't otherwise know
