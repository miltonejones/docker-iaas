## Summary

### Root Cause — three bugs conspired to lose assistant sessions on refresh:

**Bug 1: App.tsx never persisted or restored the assistant session.**  
`assistantModal` is React state initialized to `null`. On page refresh, it resets to `null`, so the assistant bar is never rendered. The `AssistantBar` component has a `sessionStorageKey` prop designed to persist the session ID to localStorage and restore it on mount — but **App.tsx never passed it**. There was also no code on mount to check for a stored session and reopen the assistant.

**Bug 2: The `beforeunload` handler saved state in the wrong format.**  
When the page unloads, the handler saved `{messages, log, pending, resolved}` directly, but `restoreFallback()` expected `{id, name, state: {messages, log, pending, resolved}, at}` (the format used by the `saveSession` catch block). So even if the fallback data was written to localStorage, it couldn't be read back.

**Bug 3: Fallback key mismatch in `restoreFallback`.**  
The `beforeunload` handler wrote to `dockyard:session-fallback:${sessionStorageKey}`, but `restoreFallback(forId)` looked for `dockyard:session-fallback:${forId}` (the actual session ID). These keys didn't match, so the fallback was invisible during restoration.

### Changes Made

**`/app/web/src/App.tsx`:**
- Added `SESSION_STORAGE_KEY = 'dockyard:assistant-session'` constant
- Added a mount effect: if `assistantPinned` is true and a session ID exists in localStorage, restores `assistantModal` to reopen the pinned assistant panel
- Passed `sessionStorageKey={SESSION_STORAGE_KEY}` to both AssistantBar instances (modal and pinned)

**`/app/web/src/components/AssistantBar.tsx`:**
- Fixed the `beforeunload` handler to save in the correct `{id, name, state, at}` format matching `restoreFallback`'s expectations
- Fixed `restoreFallback()` to try multiple key candidates — first the explicit session-ID key, then the `sessionStorageKey`-based key — so the fallback is found regardless of which path wrote it
