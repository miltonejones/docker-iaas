## Response

## Analysis & Findings

### Root Cause

The issue **iss-1784776942644-pe1l49** reported that the assistant had no voice/audio output capability — it was text-only with no TTS, audio streaming, or speech synthesis support.

### What I Found

**The feature has already been implemented** by commit `1267951 fix: Assistant has no voice/audio output capability`, which is the most recent commit on `main`. The fix is complete across three files:

#### `web/src/components/AssistantBar.tsx` — Full TTS implementation:
- **State variables** (lines 286–291): `speakingMessageIndex`, `availableVoices`, `selectedVoiceUri`, `voiceMenuOpen`, `utteranceRef`, `voiceMenuRef`
- **`loadVoices()`** (lines 514–528): Discovers browser speech synthesis voices, persists the user's voice preference via `localStorage`, defaults to the user's locale-matched voice
- **`speakMessage(index, text)`** (lines 541–592): Strips markdown for cleaner TTS output, creates a `SpeechSynthesisUtterance` with the selected voice/rate/pitch, and streams it through the browser's Web Speech API
- **`stopSpeaking()`** (lines 595–599): Cancels any in-progress speech via `window.speechSynthesis.cancel()`
- **Voice selector UI** (lines 1343–1381): A dropdown menu in the session bar listing all available system voices, with "System default" option and active-voice highlighting
- **Per-message "Read aloud" buttons** (lines 1461–1468): Each assistant message has a speak button (appears on hover) that toggles speech for that specific message

#### `web/src/icons.tsx` — Voice icons:
- `speak: faVolumeHigh` (line 75) — used for the speak/read-aloud button
- `'speak-stop': faStop` (line 76) — shown while speech is active
- `mute: faVolumeXmark` (line 77) — defined for potential use

#### `web/src/styles.css` — Voice UI styling:
- `.assistant-voice-select` / `.assistant-voice-menu` (lines 1331–1383): Positions and styles the voice selection dropdown
- `.assistant-log__speak` / `.assistant-log__speak--active` (lines 1568–1599): Styles the per-message speak button with hover reveal and active-state accent coloring

### Architecture

The feature uses the browser's built-in **Web Speech API** (`SpeechSynthesis`), which requires no server-side changes — all TTS processing happens client-side. The server (`server/src/routes/assistant.ts`) and API client (`web/src/api.ts`) needed no modifications.

### Outcome

No code changes are needed. The assistant now fully supports voice/audio output with:
- Speaking responses aloud via browser TTS
- Voice selection from all available system synthesis voices
- Per-message read-aloud controls with markdown stripping for clean speech
- Visual speaking indicator on both the message button and the session bar
- Persistent voice preference across sessions
