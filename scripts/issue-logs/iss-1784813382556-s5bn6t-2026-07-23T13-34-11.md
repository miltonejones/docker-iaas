## Summary

### Root Cause

The bug is in `web/src/components/AssistantBar.tsx` at the session-loading effect (line 804). When the user submits a new query from the toolbar search bar, the `AssistantBar` component remounts with `initialPrompt` set (the new query text) but `initialSessionId` as `undefined`. Two effects fire concurrently on mount:

1. **Session-loading effect**: Reads `localStorage.getItem(sessionStorageKey)`, finds the old session ID, and starts loading it **asynchronously**.
2. **Initial prompt effect**: Sees `initialSessionId` is `undefined`, fires `askWithText(text)`, which sends the new prompt with `rawMessages = []` to the server.

The race: `askWithText` sends the prompt correctly with empty history (the async session load hasn't completed yet), but when the async `load()` call from effect 1 eventually resolves, it **overwrites** the component state:

```typescript
setLog(session.state.log ?? []);         // ← replaces the new conversation
setRawMessages(session.state.messages ?? []);  // ← replaces with old session's data
```

The old session's full chat history replaces the new conversation that just started — that's the session bleed.

### Fix

**File:** `web/src/components/AssistantBar.tsx`, line 808–811

Added a guard to skip loading from localStorage when `initialPrompt` is provided. `initialPrompt` indicates the user is starting a fresh query from the toolbar — not resuming a saved session. The fix prevents the async session load from ever starting, so it can't race and overwrite the new conversation:

```typescript
// Before:
const savedSessionId = initialSessionId ?? (
  sessionStorageKey ? localStorage.getItem(sessionStorageKey) : null
);

// After:
const savedSessionId = initialSessionId ?? (
  sessionStorageKey && !initialPrompt ? localStorage.getItem(sessionStorageKey) : null
);
```

The explicit `initialSessionId` path (opening a saved session from the sidebar) is unaffected — it continues to take precedence via the `??` operator.
