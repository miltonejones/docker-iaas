## Diagnosis

**Root cause**: The `Instances` component in `web/src/components/Instances.tsx` rendered **all** containers at once with no pagination. The gateway routes list had already been updated with client-side pagination (`PAGE_SIZE = 10`, Prev/Next controls), but the instances list was never given the same treatment. A user with 15+ containers would see all of them in one long list with no way to navigate by page.

**Fix**: Added client-side pagination to the `Instances` component, following the exact same pattern used in `GatewayPanel.tsx` (`GatewayList`):

1. **Added `useEffect` to imports** (line 1)
2. **Added pagination state/logic** (lines 47–56):
   - `PAGE_SIZE = 10` constant
   - `page` state starting at 0
   - `totalPages` computed from `containers.length / PAGE_SIZE`
   - `effectivePage` that clamps safely when the list shrinks
   - `paginatedContainers` slice of the current page
   - `useEffect` to reset to page 0 when containers are added/removed
3. **Replaced `containers.map(` → `paginatedContainers.map(`** in both the **grid view** (line 118) and **list/table view** (line 224)
4. **Added pagination controls** (lines 319–340): Prev / "Page X of Y" / Next buttons, matching the existing `.pagination` CSS rules

No server-side changes were needed — the existing `GET /api/containers` endpoint returns all containers, and pagination is purely client-side (consistent with how gateway routes work). The CSS classes `.pagination` and `.pagination__info` already existed in `styles.css` from the gateway pagination feature.
